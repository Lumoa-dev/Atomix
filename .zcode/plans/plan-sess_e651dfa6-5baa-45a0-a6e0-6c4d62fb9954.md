## Atomix Rust 实现 — 任务文档写作计划

### 文档定位
一份同时作为**开发实施指南**和**验收契约**的任务文档，覆盖 4 个里程碑的完整需求、技术方案和验收条件。

### 文档架构

```
Atomix Rust 实现任务文档
├── 1. 项目概述（引用 01-总纲与哲学.md / 九字真言 / 全平台目标）
├── 2. 里程碑规划
│   ├── 2.1 Phase 1: 语言核心与编译器（词法/语法/语义/IR/优化/链接）
│   ├── 2.2 Phase 2: 虚拟机与指令执行（VM/52条指令/沙箱/ECALL）
│   ├── 2.3 Phase 3: 运行时与调度系统（任务池/批次/自适应控制器/OOM/滑道）
│   └── 2.4 Phase 4: 工具链与标准库（CLI/包管理/标准库/测试/调试器）
├── 3. 技术栈与工程约束
│   ├── 3.1 Rust stable / tokio / 全平台
│   ├── 3.2 unsafe 策略（核心安全 Rust，ECALL 层可控 unsafe）
│   ├── 3.3 最小依赖原则
│   └── 3.4 测试与CI
├── 4. 需求分解（核心章节）
│   └── 每条需求结构：【编号】【标题】【原文档引用】【描述】【验收标准】【优先级】【依赖】
│   ├── 4.1 Phase 1 需求（~100-120条）
│   ├── 4.2 Phase 2 需求（~80-100条）
│   ├── 4.3 Phase 3 需求（~100-120条）
│   └── 4.4 Phase 4 需求（~70-90条）
├── 5. 开发规范（分支/Commit/代码风格/代码审查）
├── 6. 验收机制
│   ├── 6.1 逐条对照原则
│   ├── 6.2 Agent 自动验收设计（对照矩阵格式/扫描策略/报告格式）
│   ├── 6.3 覆盖率计算（分母=当前里程碑P0+P1需求，分子=已实现数，合格线≥70%）

├── 7. 持续演进策略
└── 8. 附录：全量原子需求对照矩阵
```

### 写作原则
1. 每条需求精确引用原文档（文档名.md §X.Y）
2. 验收标准二进制可测（通过/不通过，Agent 可判定）
3. 粒度可独立实现和验证
4. 覆盖 docs/ 下所有 Markdown 文档（含附录）
5. 预计总需求数 350-500 条

### 篇幅估计
预计 2-3 万字，含完整原子需求清单和对照矩阵。
