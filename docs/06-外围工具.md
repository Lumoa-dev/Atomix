## 1. 分发架构

Atomix 的工具链遵循**统一入口 + 专精后端**的模型。一个安装包全装，不区分"开发机装什么、生产机装什么"。

```
atomix/                          # 安装目录
├── atomix.exe                   # 薄调度层——解析子命令，派发到后端
├── atomix-runner.exe            # 执行引擎（最小，核心）
├── atomix-build.exe             # 编译器管线
├── atomix-debug.exe             # 调试器
├── atomix-pm.exe                # 包管理
├── atomix-format.exe            # 格式化
├── atomix-lint.exe              # 代码检查
└── std/                         # 标准库 .atxo 文件
```

### 层级关系

```
atomix <子命令> [args...]
       │
       ▼
atomix.exe（薄调度层）
  ├── 解析子命令 → 查后端映射表
  ├── 找到对应 exe，传参调用
  └── 透传退出码和输出

后端 exe 映射:
  build             →  atomix-build.exe
  runner            →  atomix-runner.exe
  origin            →  atomix-runner.exe（通过 --origin 走远程协议）
  task              →  atomix-debug.exe（深度检查后端）
  test              →  atomix-runner.exe
  analysis --format →  atomix-format.exe
  analysis --lint   →  atomix-lint.exe
  analysis (其他)   →  atomix-build.exe
  add/rm/lock/sync  →  atomix-pm.exe
  cache             →  内置（atomix.exe 自管理）
  init/info         →  内置（atomix.exe 自管理）
```

用户不需要知道后端有几个 exe、分别叫什么——始终只用 `atomix <动词>`。

## 2. 代码组织：单 Package 多 Binary

### 2.1 为什么不是 Workspace 拆 Crate

Atomix 的编译器、VM、调试器、格式化器需要对**同一份 ISA 定义**（54 条 opcode 的编码模板、寄存器约定、Profile 语义矩阵）保持完全一致的理解。如果拆成独立 crate，每次改指令集就要发版本号，六个 crate 逐个升级依赖——不升级的 crate 拿到旧定义，编译照样过，但生成的行为已经不同步了。

单 package 多 binary 消除了这种同步成本：改一次 `src/base/isa.rs`，编译器自动检查所有二进制是否适配。

### 2.2 目录结构

```
Atomix/
├── Cargo.toml                    ← 一个 package, 声明 [[bin]]
├── Cargo.lock
├── src/
│   ├── lib.rs                    ← pub mod base
│   ├── base/                     ← 所有二进制共享的代码
│   │   ├── mod.rs
│   │   ├── isa.rs                ← opcode 74 条 + 编码模板 + 寄存器模型
│   │   ├── ir.rs                 ← .atxe header + section table + 各段结构
│   │   ├── atxp.rs               ← 16B 帧头 + 消息序列化
│   │   ├── vm.rs                 ← VM dispatch loop + AOT 后端
│   │   ├── compiler/             ← 编译管线 (lexer→parser→...→link)
│   │   ├── runtime/              ← 任务池 + 批次管理 + 内存墙 + 调度
│   │   ├── debug/                ← ATXP 客户端 + 断点 + 反汇编 + REPL
│   │   ├── pm/                   ← 依赖解析 + Git index
│   │   └── format/               ← 格式化引擎
│   │
│   └── bin/                      ← 每个文件 = 一个二进制入口
│       ├── atomix.rs             → atomix.exe (CLI 派发器)
│       ├── atomix-debug.rs       → atomix-debug.exe
│       ├── atomix-pm.rs          → atomix-pm.exe
│       ├── atomix-build.rs       → atomix-build.exe
│       ├── atomix-runner.rs      → atomix-runner.exe
│       ├── atomix-format.rs      → atomix-format.exe
│       └── atomix-lint.rs        → atomix-lint.exe
```

### 2.3 Cargo.toml

```toml
[package]
name = "atomix"
version = "0.1.0"
edition = "2021"

# Cargo 自动扫描 src/bin/*.rs 为独立二进制
# 也可显式声明:
# [[bin]]
# name = "atomix-debug"
# path = "src/bin/atomix-debug.rs"

[dependencies]
clap = "4"
tokio = "1"
prost = "0.12"
# ... 所有二进制共享同一份依赖树
```

`cargo build --release` → `target/release/` 下七个 exe 全出来。

### 2.4 各二进制体积

| 二进制 | 入口 | 体积 | 使用场景 |
|--------|------|------|----------|
| `atomix-runner.exe` | `src/bin/atomix-runner.rs` | 很小（VM + 调度器） | 生产常驻 + 开发执行 |
| `atomix-build.exe` | `src/bin/atomix-build.rs` | 较大（编译管线全链路） | 开发和 CI |
| `atomix-debug.exe` | `src/bin/atomix-debug.rs` | 中等 | 深度检查时 |
| `atomix-pm.exe` | `src/bin/atomix-pm.rs` | 中等 | 依赖管理时 |
| `atomix-format.exe` | `src/bin/atomix-format.rs` | 很小 | 格式化时 |
| `atomix-lint.exe` | `src/bin/atomix-lint.rs` | 很小 | 代码检查时 |
| `atomix.exe` | `src/bin/atomix.rs` | 极小（薄派发层） | CLI 入口 |

### 2.5 入口文件示例

每个 `src/bin/*.rs` 只做一件事——调 `src/base/` 里的 `run()`。不写业务逻辑。

```rust
// src/bin/atomix-debug.rs —— 全部代码
fn main() -> Result<(), atomix::base::Error> {
    atomix::base::debug::run()
}
```

体积差异来自链接器——`atomix-runner` 只链接 `base::runtime` 和 `base::vm`，`atomix-build` 会链接 `base::compiler` 全链路。未使用的 `base/` 模块被编译器死代码消除，不进入最终 exe。

## 3. 环境目录

```
.atomix/
├── cache/         # 易失——删了可完全重建
│   ├── deps/      # git clone 的依赖
│   ├── index/     # 远程索引缓存
│   └── tmp/       # 编译临时文件
├── build/         # .atxo 中间产物
├── output/        # .atxe 最终产物
└── state/         # 运行时状态
```

## 4. 完整命令参考

完整的命令树、子命令、参数表和语法规则请参见：

> **[命令行规范.md](./命令行规范.md)**

简要概览（共 13 个顶级子命令）：

| 命令 | 一句话 |
|------|--------|
| `build` | 编译 `.atx` → `.atxe` |
| `runner` | 本地执行引擎（run/status/stop） |
| `origin` | 远程连接管理（add/list/status/log/remove） |
| `task` | 深度检查 + 脚手架（add/step/mem/log/stop/restart） |
| `test` | 测试（含远程压测） |
| `analysis` | 代码质量（lint/format/type/safe/complexity/fix/check） |
| `add` | 装包 |
| `rm` | 删包 |
| `lock` | 锁定依赖 |
| `sync` | 同步 |
| `cache` | 缓存管理 |
| `init` | 项目初始化 |
| `info` | 项目/版本信息 |
