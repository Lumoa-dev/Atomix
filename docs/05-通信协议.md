# Atomix 通信协议 (ATXP)

> 版本: v0.5 (重设计)
> 最后更新: 2026-07-22
> 设计原则: 场景驱动、统一报文层、可插拔传输层
> 序列化: 固定 16B 帧头 (CRC-16-CCITT) + Protobuf 载荷

---

## 1. 概述

ATXP（Atomix Transfer Protocol）是 Atomix 工具链各组件之间的通信主干。所有跨进程交互——任务提交、状态监控、运行时检查、连接管理——均通过此协议完成。

### 1.1 通信场景

| 场景 | 参与方 | 传输 |
|------|--------|------|
| 任务提交 | CLI / PM → Runner | TCP |
| 任务监控 | Debugger / PM → Runner | TCP |
| Runner 监控 | Debugger → Runner | TCP |
| 运行时检查 | Debugger → Runner | TCP |
| 本地深度调试 | Debugger (本地模式, 进程内) | 模块复用 (非 ATXP) |

### 1.2 核心原则

- **一套消息类型覆盖全场景**：不区分"本地端点"和"远程端点"
- **帧层与载荷分离**：16B 帧头负责传输完整性，Protobuf 负责语义
- **资源/操作模型**：用 `(resource, action, params)` 替代层级路径
- **传输层透明**：同一帧可通过 TCP、Unix Socket、共享内存传递

## 2. 帧层

### 2.1 帧格式

```
Byte  0-1: magic       = 0x4154 ("AT")
Byte    2: version      = 0x01
Byte    3: msg_type     (消息类型, 见 §3)
Byte  4-7: seq_id       (u32 LE, 请求-响应匹配)
Byte    8: flags        (见 §2.2)
Byte 9-12: payload_len  (u32 LE, 不含帧头)
Byte   13: req_type     (响应帧回显原请求的 msg_type; 请求帧=0)
Byte 14-15: checksum    (CRC-16-CCITT over bytes 2-13)
Byte  16+: payload      (Protobuf 编码)
```

### 2.2 Flags 位定义

| 位 | 名称 | 说明 |
|----|------|------|
| 0 | C (Compressed) | 载荷经 zstd 压缩 |
| 1 | A (Ack Requested) | 请求对端回复 Ack |
| 2 | E (Error) | 本帧为错误响应 |
| 3 | P (is_response) | 本帧为响应帧 (req_type 有效) |
| 4-7 | — | 保留 |

### 2.3 接收流程

```
1. 读取 16B 帧头
2. 校验 magic == 0x4154; 失败 → 帧同步恢复 (以 16B 滑动搜索)
3. 校验 checksum (CRC-16-CCITT over bytes 2-13)
4. 校验 payload_len ≤ 16 MB (命令/查询) 或 ≤ 1 MB (事件)
5. 读取 payload_len 字节
6. flags & C → zstd 解压
7. 根据 msg_type 选择 Protobuf 消息解码
```

## 3. 消息类型

共 10 种消息类型。

| Type | 名称 | 方向 | 用途 |
|------|------|------|------|
| 0x01 | Hello | 双向 | 连接握手、能力协商、认证 |
| 0x02 | Ack | 双向 | 确认 |
| 0x03 | Error | 双向 | 错误报告 |
| 0x04 | Request | C → R | 通用请求 (查询/操作) |
| 0x05 | Response | R → C | 通用响应 |
| 0x06 | Event | R → C | 事件推送 |
| 0x07 | Submit | C → R | 任务提交 |
| 0x08 | SubmitResult | R → C | 提交结果 |
| 0x09 | Heartbeat | 双向 | 保活 |
| 0x0A | Bye | 双向 | 断开 |

### 3.1 与旧版对照

| 旧版 (v0.4) | 新版 (v0.5) | 变更 |
|-------------|------------|------|
| Capabilities (0x01) | 合并到 Hello | 握手一体化 |
| Command (0x04) | 合并到 Request | 操作统一 |
| Query (0x05) | 合并到 Request | 查询统一 |
| QueryResult (0x06) | 合并到 Response | 响应统一 |
| Stream (0x08) | 移除 | 后续按需加回 |
| Inject (0x09) | 合并到 Submit | 提交统一 |
| InjectResult (0x0A) | 合并到 SubmitResult | 响应统一 |
| Disconnect (0x0C) | 改为 Bye (0x0A) | 更简洁 |
| TaskOutput (0x0F) | 合并到 Response | 响应统一 |
| OutputRequest (0x10) | 合并到 Request | 查询统一 |

## 4. 请求/响应模型

### 4.1 Request

```
Request {
    resource: string      // 目标资源: "runner" | "task" | "controller" | "slots"
    action:   string      // 操作: "status" | "list" | "get" | "submit" | "config"
    params:   map<string, bytes>  // 操作参数 (key-value, value 为 Protobuf 编码)
}
```

### 4.2 Response

```
Response {
    resource: string      // 回显
    action:   string      // 回显
    data:     bytes        // 响应体 (Protobuf 编码, 类型取决于 resource+action)
    error:    string       // 非空 = 失败
}
```

### 4.3 请求-响应匹配

- 请求方分配递增 `seq_id`，响应方回显 `seq_id` + `req_type=0x04`
- 匹配条件: `seq_id` 相等 且 `flags.P == 1` 且 `req_type == 0x04`
- 超时: 30s 无响应视为失败

## 5. 端点定义

### 5.1 Runner

| Action | 参数 | 响应类型 | 说明 |
|--------|------|---------|------|
| `status` | — | RunnerStatus | Runner 健康状态 |
| `config` | — | RunnerConfig | 当前配置 |

### 5.2 Task

| Action | 参数 | 响应类型 | 说明 |
|--------|------|---------|------|
| `submit` | source/binary, name, output_mode | SubmitResult | 提交任务 |
| `list` | — | TaskList | 任务列表 |
| `status` | task_id | TaskStatus | 任务状态 |
| `output` | task_id | TaskOutput | 任务产出 |
| `stats` | task_id | TaskStats | 执行统计 |
| `regs` | task_id | RegisterSnapshot | 寄存器快照 |
| `mem` | task_id, addr, len | MemoryRegion | 内存读取 |
| `stack` | task_id | CallStack | 调用栈 |
| `is` | task_id | TaskContext | IS* 上下文 |
| `source` | task_id | SourceMap | 源码映射 |
| `segments` | task_id | SegmentData[] | 段数据 |

**限制**：`regs` / `mem` / `stack` / `is` 仅对非 Running 状态的任务可用。正在执行的任务 VM 不可查询。

### 5.3 Controller

| Action | 参数 | 响应类型 | 说明 |
|--------|------|---------|------|
| `status` | — | ControllerState | 自适应控制器状态 |

### 5.4 Slots

| Action | 参数 | 响应类型 | 说明 |
|--------|------|---------|------|
| `layout` | — | SlotLayout | 内存槽位布局 |

## 7. 连接生命周期

```
Client                              Server
  │                                   │
  │── Hello (auth_token, features) ──→│  握手 + 认证 + 能力协商
  │←── Hello (version, deny_cmds) ───│
  │                                   │
  │── Request / Submit ──────────────→│  正常通信
  │←── Response / SubmitResult ──────│
  │←── Event (push notification) ────│
  │                                   │
  │── Heartbeat ────────────────────→│  保活 (30s 间隔)
  │←── Ack ─────────────────────────│
  │                                   │
  │── Bye (reason) ─────────────────→│  断开
  │←── Bye ─────────────────────────│
```

### 7.1 Hello 握手

```
Hello {
    protocol_version: u32  (0x0001 — 主版本高16位, 次版本低16位)
    features:         u64  (能力位图)
    auth_token:       string
    client_id:        string
}

Server Hello {
    protocol_version: u32
    features:         u64  (取双方交集)
    runner_id:        string
    deny_commands:    []string
}
```

版本协商: 主版本必须一致; 次版本取 `min(client, server)`。Features 取 `client & server`(位与)。

### 7.2 Heartbeat

- 间隔: 30s 无消息时发送
- 超时: 90s 无任何消息 → 判定对端死亡, 关闭连接
- 载荷: `Heartbeat { client_time: u64 }`

### 7.3 Bye

```
Bye {
    reason:  enum { NORMAL, VERSION_MISMATCH, AUTH_FAILED, TIMEOUT, PROTOCOL_ERROR }
    message: string
}
```

## 8. Submit 流程

### 8.1 提交

```
Client → Server: Submit {
    mode:           enum { SOURCE, BINARY }
    source:         string          (mode=SOURCE 时: .atx 源码)
    binary:         bytes           (mode=BINARY 时: .atxe 字节)
    task_name:      string
    output_mode:    enum { POLL, CALLBACK, STREAM }
    callback_url:   string          (mode=CALLBACK 时)
    auth_token:     string
    priority:       u32             (0-10, 默认 5)
    timeout_ms:     u64             (0=无超时)
    compile_opts:   CompileOptions { opt_level, debug_info }
}

Server → Client: SubmitResult {
    status:         enum { ACCEPTED, REJECTED, COMPILE_ERROR }
    task_id:        string          (ACCEPTED 时)
    message:        string
    compile_errors: []CompileError  (COMPILE_ERROR 时)
}
```

### 8.2 产出获取

**POLL 模式**：客户端轮询

```
Request { resource: "task", action: "output", params: { task_id } }
→ Response { data: TaskOutput }
```

**CALLBACK 模式**：Runner 完成后 HTTP POST 到 callback_url。
**STREAM 模式**：Runner 完成后在同一条 ATXP 连接上推送 Event。

## 9. Event 推送

客户端可通过 Request 订阅事件（`action: "subscribe"`），取消订阅（`action: "unsubscribe"`）。

```
Event {
    type:       enum { TASK_DONE, TASK_ERROR, OOM_WARNING, RUNNER_STOPPED }
    task_id:    string
    timestamp:  u64
    data:       bytes
}
```

## 10. 实现范围

### Phase 1: 核心端点 (Runner Server + Client)

- `runner.status`
- `task.submit` (真实编译入池执行)
- `task.list`
- `task.status`
- `task.output`
- `controller.status`
- Hello / Heartbeat / Bye

### Phase 2: 运行时检查

- `task.regs` / `task.mem` / `task.stack`
- `task.is` / `task.stats`
- `task.source` / `task.segments`
- `runner.config`

### Phase 3: 高级功能

- `slots.layout`
- Event 订阅/推送
- CALLBACK 模式
- 安全层 (认证 / deny_commands / TLS)

## 附录: 与旧版协议的主要区别

| 方面 | 旧版 (v0.4) | 新版 (v0.5) |
|------|-----------|-----------|
| 消息类型 | 16 种 | 10 种 |
| 请求模型 | Command / Query / QueryResult 三分 | Request / Response 统一 |
| 端点路由 | 层级路径 `runner/tasks/{tid}/regs` | 资源/操作 `(task, regs, {id})` |
| 握手 | Capabilities 独立消息 | 合并到 Hello |
| 提交 | Submit + Inject 分离 | 统一为 Submit (source/binary mode) |
| 产出获取 | OutputRequest / TaskOutput 专用类型 | 合并到 Request/Response |
| Stream | 独立消息类型 | 后续按需加回 |
| 协议文档 | 2500+ 行, 混合 debug 视图定义 | ~400 行, 纯协议定义 |
