## 1. 动机

AEP-001 统一了装饰器的出现位置（数据流箭头前、函数定义前、模板名前、方法定义前），但留有三个缺口未解决：

**缺口一：装饰器只能引用 TOOLS 函数，WORKS 模板不能当装饰器。**

当前语义分析器（`semantic.rs:488-493`）校验装饰器时只查询 TOOLS 函数表，错误信息写的是"需在 TOOLS 区声明"。但 WORKS 模板在结构上与函数没有本质区别——它们有输入、有处理逻辑、有输出。没有理由禁止 WORKS 出现在 `[ ]` 位置。

**缺口二：装饰器无参数机制。**

当前装饰器语法是纯标识符 `[name]`，无法传参。这意味着装饰器的行为完全由函数逻辑硬编码，调用方无法配置。对比 Python 的 `@retry(max=3)` 或 `@json(schema="v2")`，无参数的装饰器表达能力受限。

**缺口三：装饰器的输入/输出契约是隐式的。**

当前系统不检查装饰器的输入输出类型是否与前级和后级匹配。一个接收 `bytes` 产出的 `dict` 的装饰器，被放到一个产出 `int` 的源后面，错误直到运行时才暴露。

本提案填补这三个缺口，同时确立一条核心原则：**装饰器的行为不依赖于它被用在什么上下文中。**

## 2. 设计

### 2.0 核心原则：上下文无关

装饰器（无论是 TOOLS 函数还是 WORKS 模板）的行为在所有出现位置保持一致：

- 在 `[ ]` 位置做装饰器，和在 `CALL` 位置做普通调用，行为一致
- 数据来源不同（装饰器由前级自动传入，普通调用由参数手动传入），但被调方不需要区分
- WORKS 的钩子系统（INIT、START、DONE、ERROR 等）在装饰器上下文中按相同的生命周期触发
- **不存在"在装饰器上下文下的特殊行为"**——这与指针在不同平台不同行为一样有害

### 2.1 一切皆可装饰器

任何 TOOLS 函数和任何 WORKS 模板都可以出现在 `[ ]` 位置，无需注册、无需声明、无需元信息。

编译器唯一校验的是**签名兼容性**：

```
<decorator> ::= '[' <identifier> [ '(' <param-list> ')' ] ']'
```

语义分析阶段，编译器检查：
- 被引用的标识符是已声明的 TOOLS 函数或 WORKS 模板
- 装饰器的输入类型（由前级产出决定）与目标函数/Works 的参数类型兼容
- 装饰器的输出类型与后级期望的输入类型兼容

未通过签名检查的装饰器位置产生编译错误。

### 2.2 args/kwargs 参数机制

引入两个特殊标识符 `args` 和 `kwargs`，作为装饰器/函数的隐式参数入口。

**规则：**

- `args` 和 `kwargs` 可以出现在任何 TOOLS 函数或 WORKS 方法的参数列表中
- 它们没有类型标注——`args` 自动为 `list` 类型，`kwargs` 自动为 `dict` 类型
- 写了就有，不写就没有。编译器不主动注入，不包办
- `args` 和 `kwargs` 可以与其他显式参数混写

**在装饰器上下文中的语义：**

```
TOOLS : {
    fn msgpack_decode(args, kwargs) : dict {
        CALL unpack(args[0], kwargs.schema)
    }
}

INPUT : {
    FILES : "data.bin" [msgpack_decode(schema="v1")] => data
}
```

编译器把前级数据（FILES 读出的字节）自动填入 `args[0]`，把括号中的参数 `schema="v1"` 填入 `kwargs`。

等同于：

```
WAIT msgpack_decode(input_bytes, schema="v1") => data
// 编译器将 input_bytes 填 args[0]，schema="v1" 填 kwargs
```

**在普通调用上下文中的语义：**

```
TOOLS : {
    fn process(args, kwargs) : dict {
        CALL transform(args[0], kwargs.mode)
    }
}

TASK : {
    process("data", mode="strict") => result
    // args[0] = "data"，kwargs.mode = "strict"
}
```

`args`/`kwargs` 不只在装饰器上下文有效——它是通用的参数接收机制，在函数调用中同样工作。

**装饰器参数语法：**

```
<decorator-params> ::= <ident> '=' <value> [ ',' <decorator-params> ]
```

示例：

```
[msgpack_decode(schema="v1")]
[retry(max=3, backoff="exponential")]
[DataPipeline(mode="strict", schema="v2")]
```

### 2.3 WORKS 当装饰器

WORKS 模板出现在 `[ ]` 位置时，按以下规则处理：

**数据流：**

```
FILES : "data.bin" [MyWorks] => result

// 等价于：
// 1. FILES 产出 bytes → 填入 args[0]
// 2. MyWorks 实例化：args[0] 对应该 Works 的属性/方法参数
// 3. 执行 Works 的代码体（钩子链 + PUB fn 调用）
// 4. Works 的产出 → 绑定到 result
```

**生命周期（不变）：**

Works 自身的钩子链（INIT → START → STEP → DONE → ERROR → FINALLY）按原有规则触发。它们不需要知道自己是作为装饰器还是作为 `WAIT` 调用的。

**签名匹配规则：**

| Works 状态 | 规则 |
|-----------|------|
| 有 `args` 参数 | `args[0]` 接收前级数据 |
| 无 `args` 参数但有显式参数 | 显式参数的签名必须与前级产出类型匹配 |
| 有 PUB fn 方法 | 首个 PUB fn 的返回类型决定装饰器的输出类型 |
| 无 PUB fn 方法 | Works 不产出数据，装饰器位置报错 |

**示例：**

```
WORKS DataValidator {
    PUB fn process(args, kwargs) : bool {
        CALL validate(args[0], kwargs.schema)
    }
}

INPUT : {
    FILES : "data.json" [DataValidator(schema="user_v2")] => valid : bool
}
```

### 2.4 元装饰器（预备）

元装饰器是作用于其他装饰器（或函数/Works）的编译期装饰器，采用 `#[name]` 语法与运行时装饰器 `[name]` 区分：

```
#[keyword(base=FILES)]
fn JSON(args, kwargs) : dict { ... }
```

元装饰器不参与运行时数据流，只在编译期生效。其完整规范见 AEP-006。本提案仅确立 `#[]` 语法保留给元装饰器使用。

## 3. 示例

### 3.1 函数作为装饰器

```atx
TOOLS : {
    fn gzip_decode(args, kwargs) : bytes {
        CALL gunzip(args[0])
    }

    fn json_parse(args, kwargs) : dict {
        CALL decode_json(args[0])
    }
}

INPUT : {
    FILES : "data.gz" [gzip_decode] [json_parse] => data : dict
}
```

### 3.2 带参数装饰器

```atx
TOOLS : {
    fn decrypt(args, kwargs) : bytes {
        CALL aes_decrypt(args[0], kwargs.algorithm)
    }
}

INPUT : {
    FILES : "secret.bin" [decrypt(algorithm="aes-256")] => data : bytes
}
```

### 3.3 Works 作为装饰器

```atx
WORKS Batcher {
    BATCH_SIZE : int = 100

    PUB fn process(args, kwargs) : list {
        CALL split_into_batches(args[0], self.BATCH_SIZE)
    }
}

INPUT : {
    FILES : "large.csv" [Batcher(batch_size=500)] => batches : list
}
```

### 3.4 args/kwargs 在普通调用中

```atx
TOOLS : {
    fn log(args, kwargs) {
        CALL write_to_file(args[0], kwargs.level)
    }
}

TASK : {
    log("system started", level="info")
    # args[0] = "system started", kwargs.level = "info"
}
```

### 3.5 混合显式参数与 args/kwargs

```atx
TOOLS : {
    fn process(data: bytes, kwargs) : dict {
        # data = 显式第一个参数 = args[0] 的别名
        # kwargs.mode = 配置参数
        CALL transform(data, kwargs.mode)
    }
}
```

## 4. 影响范围

| 组件 | 影响 |
|------|------|
| **词法分析** | 新增装饰器参数语法：`(` 在 `[` 后允许 |
| **语法分析** | `parse_decorators()` 扩展为支持 `[name(params)]`；`parse_decorator_params()` 新增 |
| **语义分析** | 装饰器校验范围从 TOOLS 函数扩展到 TOOLS + WORKS；新增签名兼容性检查 |
| **符号表** | `args`/`kwargs` 作为预定义标识符加入作用域（仅当用户显式写出时激活） |
| **代码生成** | 装饰器展开为 CALL 序列的逻辑不变，但需适配参数传递 |
| **VM** | 无影响（装饰器完全展开为普通指令） |
| **标准库** | 现有装饰器（如 `[gzip]`、`[encrypt]`）可受益于参数机制 |

## 5. 向后兼容性

- 现有装饰器语法 `[name]` 完全兼容，不加括号就是无参装饰器
- 现有 TOOLS 函数即使没有 `args`/`kwargs` 仍可作为装饰器使用——前提是其签名与前级产出类型匹配
- 不破坏任何现有代码语法

## 6. 未解决的问题

1. **类型推断深度**：当装饰器链多级嵌套时，编译器是否做全链类型推断？还是只有相邻级检查？
2. **WORKS 实例化开销**：Works 作为装饰器时是否每次调用都实例化？是否做实例复用？
3. **kwargs 的类型安全**：`kwargs` 是 `dict`，键值对的类型是否要做更严格的约束（如只允许某些 key）？
