## 1. 动机

v1 语法规范仅在 INPUT/OUT 数据流箭头前定义了装饰器语法 `[identifier]`
（见 通用语法.md §18）。但装饰器模式在语言中具有通用性，自然地也应出现在：

- TOOLS 区函数定义前
- WORKS 模板定义前
- WORKS 方法定义前

当前设计未覆盖这些位置，构成语法缺口。

## 2. 设计

### 2.1 语法

装饰器的统一语法为：

```
decorator ::= '[' identifier ']'
decorators ::= decorator*
```

装饰器出现在以下位置的**最前面**：

| 位置 | 示例 | 语义 |
|------|------|------|
| 数据流箭头前 | `HTTP : "url" [gzip] [encrypt] => RAW : bytes` | 装饰器在源数据被读取后、绑定到变量前依次链式处理 |
| 函数定义前 | `[rate_limit] [log] fn handler() { ... }` | 对 TOOLS 函数的增强包装 |
| WORKS 模板名前 | `[validator] WORKS DataProc { ... }` | 对整个 WORKS 模板的装饰 |
| 方法定义前 | `[auth] PUB fn get_name() : str { ... }` | 对 WORKS 内指定方法的装饰 |

### 2.2 编译期语义

- 装饰器引用 TOOLS 区定义的函数（语法层面仅验证标识符合法性）
- 同一位置的多装饰器按从左到右的顺序链式应用
- 装饰器 `[id]` 中的 `id` 必须为简单标识符（不接受表达式 `[a + b]`）

### 2.3 运行时语义

运行时期望的语义（**Phase 3/4 时定义具体实现**）：

```
# 函数装饰器的效果等价于：
# decorated_fn(args) = decorator1(decorator2(fn(args)))
```

数据流装饰器的运行时语义已在 INPUT语法.md / OUT语法.md 中定义，此处不做变更。

## 3. 示例

```atx
TOOLS : {
    [rate_limit(max=100)]
    [log]
    fn fetch_data(url : str) : bytes {
        # ...
    }
}

[validate_token]
WORKS DataProc {
    RAW : bytes

    [auth]
    PUB fn get_name() : str { self.NAME }

    START :: process
}
```

## 4. 影响范围

| 组件 | 影响 | 工作量 |
|------|------|--------|
| 词法分析 | 无影响（`[` / `]` 已有 Token） | 0 |
| 语法分析 | 新增 `parse_decorators()` 共用函数，在 `parse_func_def()` / `parse_works_def()` 中调用 | 小 |
| 语义分析 | 验证装饰器标识符对应已注册的 TOOLS 函数 | 小 |
| IR 生成 | 函数装饰器：生成包装调用序列；WORKS 装饰器：按装饰器包装 START 方法 | 中 |
| VM 指令 | 无影响（装饰器编译为普通 CALL 序列） | 0 |
| 运行时 | 无影响 | 0 |

## 5. 向后兼容性

- **完全兼容**。v1 的数据流装饰器语法不变
- 函数/WORKS 装饰器是新增语法位置，不影响现有代码
- 已有代码无需修改

## 6. 未解决的问题

1. **装饰器参数**：`[rate_limit(max=100)]` 这种带参数的装饰器语法在 v1 中未定义。
   可能的方向：`[name(key=val, ...)]` — 提案 AEP-002 可专门讨论。
2. **装饰器对 WORKS 方法的语义**：装饰器包装方法的返回值还是整个方法调用？
   推荐按 Python 风格 — 装饰器接收方法函数并返回包装后的函数。
3. **链式装饰器执行顺序**：`[a][b]` 是先 a 再 b（从外到内），还是先 b 再 a（从内到外）？
   推荐 `[b]` → `[a]` 的从外到内（类似 Python：`@a @b` 等价于 `a(b(fn))`，
   注意源码书写顺序 `[a][b]` 中 a 是外层的，所以实际效果等价于 a(b(fn))）。
4. **编译期装饰器展开与调试信息的映射**：展开后的 CALL 序列对应的源码位置需要正确处理。
