## 1. 动机

当前 `atomix build` 每次执行都完整编译所有源文件。随着项目规模增长（多个 `.atx` 文件、依赖标准库），完整重编的耗时线性增长。增量编译的目标是：

- 只重编有变动的源文件及其传递依赖
- 无变动时毫秒级完成构建检查
- 开发者体验接近"保存即刷新"

设计文档中已定义 `.atomix/cache/` 目录的存在，但未规定具体算法。

## 2. 设计

### 2.1 缓存键

每个 `.atxo` 中间产物的缓存键由以下内容哈希决定：

```
cache_key = SHA256(
    source_file_content +
    dependencies_hash +
    compiler_version +
    opt_level
)
```

- `source_file_content`：源文件字节内容
- `dependencies_hash`：所有直接依赖 `.atxo` 的缓存键按文件名排序后拼接再哈希
- `compiler_version`：当前 `atomix --version` 输出，确保编译器升级后全部重编
- `opt_level`：优化级别（`"0"`, `"1"`, `"2"`, `"s"`）

### 2.2 目录结构

```
.atomix/cache/
├── deps/              # git 克隆的依赖（已有定义）
├── index/             # 远程索引缓存（已有定义）
├── tmp/               # 编译临时文件（已有定义）
└── objects/           # 新增：.atxo 缓存对象
    ├── ab3f2c1e.atxo  # 文件名 = cache_key 前 8 位
    ├── f7e8d2c3.atxo
    └── manifest.json  # 源文件 → cache_key 映射
```

### 2.3 manifest.json

```
{
  "version": 1,
  "compiler": "atomix 0.1.0",
  "files": {
    "tasks/cleanup.atx": {
      "cache_key": "ab3f2c1e",
      "dependencies": ["tools/utils.atxo"],
      "last_modified": "2026-07-19T10:30:00Z"
    },
    "tools/utils.atx": {
      "cache_key": "f7e8d2c3",
      "dependencies": [],
      "last_modified": "2026-07-19T10:29:00Z"
    }
  }
}
```

### 2.4 构建流程

```
atomix build
  │
  ├→ 1. 读取 manifest.json
  ├→ 2. 遍历 src/ 下所有 .atx 文件
  ├→ 3. 对每个文件：
  │     ├→ 计算 cache_key
  │     ├→ 在 .atomix/cache/objects/ 中查找
  │     ├→ 找到且 manifest 中依赖哈希匹配 → 命中缓存，跳过
  │     └→ 未命中 → 编译该文件 → 写入 cache + 更新 manifest
  ├→ 4. 对所有有变动的文件拓扑排序
  ├→ 5. 按依赖顺序增量编译
  ├→ 6. 链接最终 .atxe
  └→ 7. 写入 .atomix/output/
```

### 2.5 缓存失效

| 场景 | 行为 |
|------|------|
| 源文件内容变 | cache_key 变 → 重新编译 |
| 依赖链中的文件变 | 传递依赖哈希变 → 级联重编 |
| compiler 升级 | compiler_version 变 → 全部重编 |
| `--opt` 参数变 | opt_level 变 → 全部重编 |
| 删除 `.atomix/cache/` | 全部重编（安全） |
| manifest.json 丢失 | 全部重编（安全退化） |

### 2.6 清理策略

```
atomix clean           → 删除 .atomix/build/ 和 .atomix/output/
atomix clean --cache  → 额外删除 .atomix/cache/（下次构建全部重编）
```

LRU 淘汰策略可选：当 `.atomix/cache/` 超过 500MB 时，删除最近 30 天未访问的缓存对象。

## 3. 示例

```bash
# 首次构建（全量）
atomix build                         # 编译所有 .atx，耗时 2s

# 无变动构建（增量）
atomix build                         # 全部命中缓存，耗时 0.05s

# 修改一个文件后
echo "fn helper() {}" >> tools/utils.atx
atomix build                         # tools/utils.atx + 依赖它的文件重编
                                     # tasks/cleanup.atx 等不受影响
```

## 4. 影响范围

- 编译器：新增缓存读写逻辑，链接器需支持增量段合并
- CLI：`atomix build` 新增 `--no-cache` 参数跳过缓存
- 项目管理：manifest.json 的读写与版本兼容

## 5. 向后兼容性

完全向后兼容。不设缓存的 `atomix build source.atx` 单文件模式保持不变。
增量编译只影响 `atomix build`（无参数）项目级构建模式。

## 6. 未解决的问题

- 跨会话缓存共享（CI/CD 场景）是否需要？
- manifest.json 的并发写入安全（多个 atomix 进程同时构建）？
- 间接调用（通过 `CALL TOOLS :: func()` 跨文件引用）的依赖追踪是否需要更精确？
