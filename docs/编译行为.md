# Atomix 编译行为

> 架构版本: v0.1 (设计阶段)
> 最后更新: 2026-07-15

---

## 1. 五区结构

Atomix 源码在逻辑上由五个基础区域构成，外加文件头 USE 声明：

```
┌─────────────────────────────┐
│  文件头 USE 声明             │  引入外部依赖（不属于任何区）
├─────────────────────────────┤
│  TOOLS 区                   │  工具函数、泛型、异常定义
├─────────────────────────────┤
│  INPUT 区                   │  数据来源声明，产出文件级常量（不可变）
├─────────────────────────────┤
│  TASK 区                    │  编排逻辑，引用 INPUT 常量，产出 OUT 变量
├─────────────────────────────┤
│  WORKS 区                   │  可复用任务模板（0~N 个）
├─────────────────────────────┤
│  OUT 区                     │  数据交付，引用 TASK 声明的产出变量
└─────────────────────────────┘
```

**各区约束：**

| 区/声明 | 数量 | 说明 |
|---------|------|------|
| `USE`（文件头） | 任意 | 不属于任何区 |
| `TOOLS` | 有且仅有一个 | 工具函数、类型定义 |
| `INPUT` | 有且仅有一个 | 数据来源 |
| `TASK` | 有且仅有一个 | 编排核心 |
| `WORKS` | 任意（0~N） | 可复用任务模板 |
| `OUT` | 有且仅有一个 | 数据交付 |

### 1.1 数据流向

数据在各区之间单向流动，不存在反向引用：

```
TOOLS ──┐
         ├──► INPUT ──► TASK ──► OUT
WORKS ──┘
```

- **TASK 区**可以引用 INPUT 区定义的常量（通过 `INPUT : <常量名>`）
- **OUT 区**可以引用 TASK 区中 `OUT :` 声明的产出变量
- **TASK 区**可通过 `WAIT` 派发 WORKS 实例
- **所有区**均可引用 TOOLS 区定义的函数、泛型和异常
- 反向引用不存在——TASK 不能引用 OUT，INPUT 不能引用 TASK

### 1.2 为什么 `OUT :` 写在 TASK 区？

`OUT :` 出现在 TASK 区而不是 OUT 区，是因为职责边界不同：

| 区 | 负责 | 不负责 |
|----|------|--------|
| TASK | 计算、编排、决定"产出什么" | 不关心产出怎么交付 |
| OUT | 将产出发送到目标地址 | 不关心产出是怎么算出来的 |

`OUT : batch_data : list` 的意思是"我 TASK 算完了，声明 `batch_data` 这个变量是对外可见的产出"。这个声明是**计算的结果**，所以它属于 TASK。

OUT 区只做一件事：引用这些已声明的变量，决定怎么把它们送出去：

```
// TASK 区：声明产出
CALL : process(INPUT : RAW) => OUT : result

// OUT 区：决定交付方式
result => HTTP : "https://api.com/upload" (method=POST)
result => JSON : "/tmp/backup.json"
```

## 2. 编译期重排

编译器在编译期间会对源码进行**五区重排**，不论用户在文件中以什么顺序书写，编译器都会重排为固定顺序：

```
// 用户的书写顺序（任意）：
TASK : { ... }
OUT : { ... }
INPUT : { ... }
TOOLS : { ... }
WORKS : { ... }

// 编译器重排为：
TOOLS : { ... }
INPUT : { ... }
TASK : { ... }
WORKS : { ... }
OUT : { ... }
```

**重排后的顺序是固定的**（TOOLS → INPUT → TASK → WORKS → OUT），但书写顺序完全自由。

这使得：
- **用户书写自由**——可按直觉组织代码，不强制遵守物理顺序
- **依赖关系清晰**——编译器静态分析依赖图，确保没有循环引用
- **数据流确定**——从上到下，单向、无环、可预测

## 3. 执行顺序

编译重排之后，运行时按以下阶段执行：

1. **TOOLS 区**：加载工具函数、泛型、异常定义（全局可见）
2. **INPUT 区**：解析数据来源，产出文件级常量（不可变）
3. **TASK 区**：引用 INPUT 常量，执行编排逻辑（CALL/IF/FOR/JOIN/WAIT）
4. **WORKS 区**：按需实例化并执行任务模板
5. **OUT 区**：引用 TASK 中 `OUT :` 声明的变量，执行数据交付

TASK 区产生的变量，除显式 `OUT :` 声明的之外，全部在 TASK 结束时销毁回收。

---

## 4. 顺序的推导

五区的固定顺序（TOOLS → INPUT → TASK → WORKS → OUT）不是随意定的，每一步都有设计上的因果链。

### 为什么 TOOLS 必须最先加载

TOOLS 定义的是工具函数、泛型、异常。这些是**全文件共享的基础设施**。只有 TOOLS 加载完毕，后面的区才能引用其中的函数和类型。

这也解释了为什么**装饰器不需要写 `TOOLS::` 前缀**——TOOLS 的函数一旦加载，就自动注册到全局可见的命名空间中，所有区都能直接用函数名调用。

```
// TOOLS 定义
TOOLS {
    bytes : fn compress(bytes data) { ... }
}

// 其他区直接使用，不需要 TOOLS::compress
INPUT :
    HTTP : "url" [compress] => DATA : bytes   // compress 就是 TOOLS 里的函数
```

### 为什么 INPUT 跑完就卸载

INPUT 区是流式执行的。它的职责是"把数据从外部接进来，产出常量"——干完就撤。

INPUT 执行完毕后，以下内容全部卸载：
- INPUT 区内的临时变量、中间结果
- 网络连接、文件句柄等 I/O 资源
- 所有非 `OUT :` 产出的局部数据

**唯一保留的**：`INPUT` 区产出的文件级常量（不可变）。其他一切全部清理。

这就强迫你必须提前把所有数据的来源在 INPUT 区说清楚——INPUT 跑完就没了，你不可能在 TASK/WORKS 里回头再去拉数据。

```
INPUT :
    HTTP : "https://api.com/data" => RAW : bytes   // 常量，保留
    // 跑完后 HTTP 连接断开、临时缓冲区清理
    // 只保留 RAW 这个常量给 TASK 用

TASK :
    // 此处已经无法再发起 HTTP 请求
    // 只能用 INPUT 留下的常量
```

### 为什么 TASK 和 WORKS 不能当装饰器

装饰器 `[func]` 的本质是数据流转发函数——数据在源和目标之间经过装饰器处理。但 TASK 和 WORKS 的定位是编排和计算，它们不参与 I/O 数据流转发。只有 TOOLS 中的纯函数适合做这种"中间处理"。

### 为什么 OUT 的装饰器可以直接用

OUT 区执行时，TOOLS 早已加载完毕（顺序 1），装饰器函数已经全局可见。所以 OUT 区可以直接在装饰器位置引用 TOOLS 函数，不需要额外导入。

```
OUT :
    DATA => HTTP : "url" [encrypt]    // encrypt 是从 TOOLS 来的
```

### 总结

| 序号 | 区 | 执行后 | 设计含义 |
|------|------|--------|----------|
| 1 | TOOLS | 函数全局注册 | 装饰器不需前缀 |
| 2 | INPUT | 卸掉所有临时资源，仅保留常量 | 数据必须在 INPUT 说清楚 |
| 3 | TASK | 产出 OUT 变量，其余销毁 | TASK 和 WORKS 不做 I/O |
| 4 | WORKS | 实例按生命周期回收 | 纯计算，无副作用 |
| 5 | OUT | 交付完成，整体结束 | 装饰器直接引用 TOOLS |
