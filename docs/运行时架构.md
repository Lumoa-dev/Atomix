# Atomix 运行时架构

> 架构版本: v0.2 (机制层)
> 最后更新: 2026-07-16
> **算法细节已拆分至 [策略模块.md](策略模块.md)，仿真验证代码见 `sim/`**

## 1. 概述

运行时引擎负责 IR 加载、指令执行、任务调度和资源管理。核心设计围绕一个命题：**在资源缝隙中跑出最大吞吐量**。

## 3. 任务池

### 3.1 基本设计

任务进入系统后**不直接进内存**——先进入任务池。任务池位于磁盘上，是任务的第一站。

```
新任务到达
    │
    ▼
┌─────────────────────────────────────────────┐
│              任务池 (Task Pool)               │
│                                             │
│  每个任务条目记录：                              │
│    · task_id        任务唯一标识               │
│    · disk_addr      任务在磁盘上的存储地址       │
│    · mem_addr       任务在内存中的地址（运行时填充）│
│    · status         状态码（整数范围）           │
│    · counter        执行计数（可选）             │
└─────────────────────────────────────────────┘
```

### 3.2 双地址管理

每个任务有两个地址：

| 地址 | 含义 | 生命周期 |
|------|------|----------|
| **disk_addr** | 任务在磁盘上的持久存储位置 | 从任务进入池到任务执行完毕卸载 |
| **mem_addr** | 任务被加载到内存后的运行时地址 | 仅在 RUNNING 状态期间有效 |

任务执行完毕后，mem_addr 回收，disk_addr 标记为可清理。

### 3.3 状态码

状态码使用整数范围，每个范围代表一个生命阶段：

```
0x0000–0x0FFF    等待阶段（INIT / READY / QUEUED）
0x1000–0x1FFF    执行阶段（RUNNING / SUSPENDED / OOM_PAUSE）
0x2000–0x2FFF    完成阶段（DONE / ERROR / TIMEOUT）
0x3000–0x3FFF    回收阶段（CLEANUP / DISPOSED）
```

状态码范围化设计的好处：批次管理器无须精确知道每个任务的微观状态——它只需知道任务在哪个阶段，据此决定调度行为。

## 4. 批次管理器

批次管理器决定**同时有多少任务在内存中执行**。

### 4.1 双重上限

```
硬上限 H = min( CPU_limit, MEM_limit, IO_limit, NET_limit )
         ↑ 物理天花板——超过就 OOM

软上限 S = H × β(pool_depth)
         ↑ 吞吐量最优值——不是能不能，而是该不该

批次额度 N_batch = min(H, S)
```

### 4.2 硬上限 H：多元加权公式

H 是四条资源维度的最小值——**哪个先到瓶颈，哪个就是天花板**：

```
H = ⌊ min( C, M, I, N ) ⌋

C = (CPU_cores × α_cpu) / CPU_per_task
M = (MEM_free  × α_mem) / MEM_per_task
I = (IOPS_avail × α_io) / IOPS_per_task
N = (NET_avail  × α_net) / NET_per_task
```

硬件参数通过 OS 查询获取，每任务资源估算和保留系数 α_* 可通过 Daemon 配置覆盖（详见 [配置设计.md](配置设计.md)）。

### 4.3 软上限 S 与因子调节

S 在 H 的基础上受四个动态因子调节：积压（β）、速度（λ）、体积（σ）、方差（γ）。**因子的具体计算公式、平滑化方法和合并策略已拆分至 [策略模块.md](策略模块.md)。**

其中关键设计约束：

- **所有因子使用连续 S 型函数**，避免离散分段在边界处的 N_batch 跳变振荡
- **因子合并采用加权几何平均**，避免乘法链在多因子同时偏低时的过度收缩
- **OOM 反馈使用 AIMD + 滞回区**，避免恢复过程中再次触发 OOM 的振荡循环

四象限分析（速度 × 体积）的概念框架保留，但精确的因子取值由策略模块的连续函数决定。

### 4.4 OOM 反馈调节

当任务执行中发生 OOM 挂起时，自适应控制器介入调整 α_mem（从而影响 M 和 H）。**具体算法（AIMD 参数、滞回阈值、恢复速率）详见 [策略模块.md §7](策略模块.md)。**

### 4.5 算例

算例展示了公式框架的使用方式。精确数值随策略模块的连续函数而变化，此处保留结构说明。参见 `sim/reports/` 中的仿真结果获取不同场景下的实际 N_batch 取值。

## 5. 内存管理模型

### 5.1 墙式预分配

内存分配采用**墙式预分配**（类比 OS 的虚拟内存管理）：批次管理器先计算一个内存总池，再按批次额度均分为虚地址。**内存不实际占用，直到任务被加载执行。**

```
内存计算流程:

1. 总可用内存 = OS 报告空闲物理内存 × α_mem
        ↓
2. 任务内存池 = 总可用内存 - 安全冗余额度
        ↓
3. 每任务虚地址空间 = 任务内存池 / N_batch
        ↓
4. 虚地址空间预分配给 N_batch 个槽位（不实际占用物理内存）
```

### 5.2 一个萝卜一个坑

```
批次槽位 (N_batch 个):
┌────┬────┬────┬────┬────┬────┬────┬────┐
│ S0 │ S1 │ S2 │ S3 │ S4 │ S5 │ ... │ Sn │  每个槽位 = 虚地址空间
└────┴────┴────┴────┴────┴────┴────┴────┘
  │    │    │
  ▼    ▼    ▼
任务池 ──────► 批次管理器拉取任务 → 填入空槽位 → 分配实地址 → 执行

执行流程:
  1. 批次管理器从任务池取一个任务
  2. 找到一个空槽位 → 将虚地址映射到实地址
  3. 执行引擎开始执行
  4. 任务完成 → 卸载 → 回收实地址 → 槽位变空
  5. 新任务填入空槽位 → 循环
```

关键特性：
- **虚地址预分配不占物理内存**——N_batch 个槽位只是一个管理结构
- **实地址按需分配**——只有正在执行的任务才占用物理内存
- **槽位循环复用**——任务完成立即释放，新任务立即填入

### 5.3 安全冗余额度

内存池计算时预留的安全冗余：

```
安全冗余 = 总内存池 × β_safety    （默认 β_safety = 0.15）

作用:
  · 宿主进程自身的内存需求
  · 任务执行期间可能的内存波动
  · OOM 恢复时的扩展开销
```

### 5.4 任务 OOM 处理

任务执行过程中内存需求超出分配的虚地址空间时：

```
1. 任务不崩溃——进入 SUSPENDED 状态
2. 内存管理器重新分配更大的虚地址空间（扩容）
3. 任务恢复执行（RESUMED）
```

**不是每次 OOM 都会导致真正的暂停**——因为虚地址分配是动态的，内存管理器可以在探测到接近上限时主动扩容。

### 5.4.1 俄罗斯方块扩容（滑道机制）

槽位布局中始终保留一段**空白滑道**（大小为 1.5× max_slot 以上，由策略模块动态调整）。OOM 发生时：

```
[Slot0][Slot1][Slot2][Slot3][==== 滑道 ====]
                 ↑ OOM
                 
[Slot0][Slot1][死区][Slot3][Slot2'====滑道==]
                               ↑ 任务滑入滑道

→ 等 Slot1 或 Slot3 完成后，死区与邻居合并，重新切分槽位
```

![俄罗斯方块滑道机制](../.assets/slipway_mechanism.png)

滑道大小的动态调节算法详见 [策略模块.md §8](策略模块.md)。

### 5.5 风险管控：OOM 反馈机制

当 OOM 扩容行为频繁发生时，自适应控制器自动介入调整 α_mem 和 N_batch。反馈回路的完整算法（AIMD + 滞回）详见 [策略模块.md §7](策略模块.md)。

关键机制不变：OOM → α_mem 降低 → M 降低 → H 降低 → N_batch 降低 → 每任务内存配额增大 → OOM 减少 → α_mem 逐步恢复。

## 6. 高积压模式

### 6.1 触发条件

当任务池积压深度达到批次额度的 **3 倍以上** 时，进入高积压模式：

```
backlog_depth ≥ N_batch × 3  →  高积压模式
```

### 6.2 行为变化

高积压模式下，不再维持"一个萝卜一个坑"的连续填充策略，改为**批量—周期**调度：

```
1. 等待空余槽位达到总批次数量的 15%–30%
        │
2. 一次性从任务池中批次拉取对应数量的任务
        │
3. 批次执行这批任务
        │
4. 任务完成 → 槽位释放
        │
5. 回到步骤 1（等待再次达到 15%–30% 空余）
```

这个策略的核心收益：**减少上下文切换频率**。在高积压场景下，频繁的单个任务进出槽位会导致大量切换开销。攒够一批再执行，每次执行周期的有效工作量更大。

### 6.3 退出条件

当积压深度降到 `N_batch × 2` 以下时，退出高积压模式，回到标准模式。

## 7. 执行引擎

### 7.1 主循环

```
while (running) {
    task = batch_manager.next_ready_task();  // 取下一就绪任务
    if (task == NULL) {
        wait_for_io_or_backlog();            // 等待 IO 或新任务
        continue;
    }

    context = load_context(task);            // 恢复寄存器上下文
    execute_quantum(context, QUANTUM);       // 执行一个时间片
    save_context(task, context);             // 保存寄存器上下文

    if (task->state == DONE) {
        batch_manager.release_slot(task);    // 释放槽位，回收实地址
        task_pool.update_status(task, CLEANUP);
    } else if (task->state == OOM_PAUSE) {
        memory_manager.expand(task);         // 扩容虚地址
        task->state = RESUMED;
    } else {
        batch_manager.requeue(task);         // 未完成，放回就绪队列
    }
}
```

### 7.2 时间片

| 参数 | 默认值 | 说明 |
|------|--------|------|
| QUANTUM | 1000 条指令 | 单次连续执行的最大指令数 |
| 抢占模式 | 协作式 | 仅在 ECALL（阻塞型）、OOM 暂停和 QUANTUM 耗尽时切换 |

### 7.3 上下文切换开销

| 操作 | 开销 | 说明 |
|------|------|------|
| 寄存器保存/恢复 | ~20 条 mov 指令 | 16 × 64 位寄存器 |
| 栈切换 | 指针替换 | sp、fp 寄存器 |
| PC 恢复 | 寄存器写入 | 从保存的指令偏移恢复 |

## 8. 依赖图调度

编译器在 `.task` 段中预计算依赖图层级。调度器按层级从深到浅分批执行：

```
层级 3（最深，无依赖）: [D, E, F]  → 首批并发执行
层级 2（等子任务）   : [B, C]     → D/E/F 全部完成后执行
层级 1（根）         : [A]        → B/C 全部完成后执行
```

依赖图层级调度与批次管理独立运作——批次管理器决定"同时跑几个"，依赖图决定"先跑谁后跑谁"。

## 9. 持久运行与热注入

### 9.1 Daemon 模式

生产模式下，`atomix-daemon` 常驻运行。启动后：

1. 读取配置文件（默认 `/etc/atomix/daemon.toml`）
2. 初始化任务池（扫描 `task_dir` 中的 `.atxe` 产物）
3. 初始化批次管理器（检测硬件、计算 N_batch）
4. 监听端口（接受 `atomix deploy` 推送的新任务）
5. 进入执行循环

### 9.2 任务注入协议

```
Compiler ──► Daemon (TCP)

消息格式: [Header: 8B] [IR Binary: variable]

Header:
  byte[0-3]  Magic:  0x41545844 ("ATXD")
  byte[4-5]  Version: IR 版本号
  byte[6]    Mode:    0=替换已有任务, 1=追加为新任务
  byte[7]    Flags:   保留

Daemon 响应:
  byte[0-1]  Status:  0=成功入池, 1=格式错误, 2=池满
  byte[2-5]  TaskID:  分配的任务 ID（仅成功时有效）
```

注入的任务进入任务池（写入磁盘），按 §3–§6 的流程参与调度。

## 10. 错误传播

### 10.1 系统调用错误映射

ECALL 返回的错误码按以下规则映射为 DSL 级异常：

| ECALL 返回 | 含义 | DSL 异常 |
|-----------|------|----------|
| `err < 0` | 系统调用失败 | 对应异常类型（IOError / NetworkError 等） |
| `err == 0` | 成功 | 正常返回 |
| `err == -EAGAIN` | 资源暂不可用 | 任务挂起，等待重试 |

### 10.2 任务级异常传播

```
子任务抛出异常 → TASK_RET 返回错误码 → 父任务 JOIN 收到错误
→ 父任务可选 TRY 捕获或继续向上传播
```

## 11. 沙箱隔离

- 所有内存访问通过 LOAD/STORE 指令经边界检查
- 任务不可访问其他任务的地址空间
- ECALL 调用经过权限检查（`.task` 段声明的能力列表）
- 任务池的磁盘隔离：每个任务只能访问自己的 disk_addr 范围
