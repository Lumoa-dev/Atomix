# Atomix 包管理

> 架构版本: v0.1 (设计阶段)
> 最后更新: 2026-07-16

---

## 1. 设计原则

**无中心化注册表。** 依赖来源为 Git 仓库或本地路径。包发现通过一个 Git 仓库托管的索引文件完成——不维护服务器、不建数据库。

**一个项目管理 N 个独立 Task。** 每个 Task 是独立的编译和链接单元，各自拥有独立的依赖树。Task 之间共享项目级源码，但产出的 `.atxe` 互不依赖。

---

## 2. 项目配置：`atomix.toml`

项目根目录下的 TOML 配置文件，声明项目元信息、依赖来源和 Task 入口：

```toml
# atomix.toml

[project]
name = "my_infra"
version = "1.0.0"

# 依赖声明
[dependencies]
http = "github:atomix-eco/http@v1.0.0"
db   = "git:https://gitlab.internal.com/atomix/db.git@v2.0.0"
fs   = { path = "./vendor/fs" }

# 本地路径别名（atomix index --add 写入）
[aliases]
utils = "~/projects/shared/utils"

# Task 声明——每个 Task 一个入口文件
[[tasks]]
name = "data_cleanup"
entry = "tasks/cleanup.atx"
schedule = "0 3 * * *"

[[tasks]]
name = "log_archive"
entry = "tasks/archive.atx"
schedule = "manual"

# 构建设置
[settings]
atomix_version = "1.0"

[settings.lint]
unused_import = "warn"

[settings.test]
timeout = "30s"

# 运行时硬件限制（可被 atomix run 的临时参数覆盖）
[settings.runtime]
max_memory = "256MB"

# 部署目标
[deploy]
targets = [
    { task = "data_cleanup", host = "prod-1:9000" },
    { task = "log_archive",  host = "prod-1:9000" },
]
```

### 2.1 依赖来源格式

| 格式 | 说明 |
|------|------|
| `github:<owner>/<repo>@<tag>` | GitHub 仓库，tag = 版本 |
| `github:<owner>/<repo>#<commit>` | GitHub 仓库，精确 commit |
| `git:<url>@<tag>` | 任意 Git 仓库 |
| `{ path = "<path>" }` | 本地路径（相对或绝对） |

## 4. 环境目录：`.atomix/`

编译和运行的本地工作目录，不提交到版本控制：

```
project/
├── atomix.toml
├── atomix.lock
├── tasks/                # Task 源文件
│   ├── cleanup.atx
│   └── archive.atx
├── tools/                # 项目级 TOOLS / WORKS
│   └── utils.atx
└── .atomix/              # 环境目录
    ├── cache/            # 垃圾桶——删了可完全重建
    │   ├── deps/         # git clone 下来的依赖
    │   ├── index/        # 远程索引缓存（atomix install index）
    │   └── tmp/          # 编译临时文件
    ├── build/            # .atxo 中间产物
    ├── output/           # .atxe 最终产物（部署单元）
    └── state/            # 运行时状态（Daemon 写入）
```

| 目录 | 性质 | 说明 |
|------|------|------|
| `cache/` | 易失 | 删掉后下次 `atomix build` 可完全重建。仅用于加速 |
| `build/` | 半持久 | `atomix clean` 清除。中间编译产物 `.atxo` |
| `output/` | 持久 | 最终链接产物 `.atxe`。这是交付物 |
| `state/` | 运行时 | Daemon 写入的运行状态和队列信息 |

## 5. 依赖解析

### 5.1 远程依赖

```
atomix install index              # 首次：clone 远程索引仓库 → .atomix/cache/index/
atomix install index --update     # 后续：git pull 增量更新
atomix install http               # 查索引 → 找到源地址 → git clone → 编译为 .atxo
atomix install http@v1.0.0        # 指定版本
```

索引仓库本身是一个 Git 仓库，包含 `index.json`：

```json
{
  "packages": {
    "http": {
      "source": "github:atomix-eco/http",
      "versions": ["1.0.0", "1.1.0"],
      "description": "HTTP client/server for Atomix",
      "tags": ["network", "http"]
    }
  }
}
```

默认索引仓库为 `github:atomix-eco/index`，可通过参数覆盖：

```bash
atomix install index --source gitlab:my-company/index    # 私有索引
```

索引缓存放在 `.atomix/cache/index/`，属于垃圾桶——删掉不影响功能，只是下次需要重新 `atomix install index`。

### 5.2 本地别名

`atomix index --add` 注册本地路径别名，**严格项目级**——写入当前项目的 `atomix.toml` 的 `[aliases]` 段，不跨项目：

```bash
atomix index --add ~/projects/shared/utils --as utils
atomix index --add ./vendor/fs --as fs
atomix index --list                  # 列出当前项目的所有别名
atomix index --remove utils          # 移除别名
```

别名注册后，源码中可直接用别名引用：

```
FROM "utils" USE "helpers" :: TOOLS :: clean as clean_data
```

## 6. 包结构

一个 Atomix 包是一个目录，包含 `.atx` 源文件和项目配置：

```
my_package/
├── atomix.toml          # 项目配置（必需）
├── package.atx          # 包入口（可选——纯库包不需要入口）
├── tools/               # TOOLS 区定义
│   └── transforms.atx
├── works/               # WORKS 模板
│   ├── processor.atx
│   └── validator.atx
└── tasks/               # TASK 编排
    └── main.atx
```

包不需要显式的入口文件——如果它只是一个被引用的工具库（纯 TOOLS + WORKS），永远不需要"被执行"，则入口声明不存在。

## 7. 多文件编译模型

### 7.1 三阶段编译

```
.atx 源文件          .atxo 中间产物         .atxe 最终产物
──────────          ────────────           ────────────
main.atx    ──┐     main.atxo    ──┐
               │                    │
tools.atx   ──┤     tools.atxo   ──┤
               ├──►  ...         ──┼──►  task_name.atxe
works/       ──┤     works/      ──┤
  proc.atx     │       proc.atxo    │
               │                    │
依赖 (.atxo)  ──┘                    │
(from cache)                        ┘
```

| 阶段 | 输入 | 输出 | 做什么 |
|------|------|------|--------|
| **编译** | `.atx` 源文件 | `.atxo` 中间文件 | 词法→语法→IR 生成，每个文件独立编译。外部引用保留为符号占位 |
| **链接** | N × `.atxo` + 依赖 | `.atxe` 最终产物 | 符号解析、段合并、依赖图拓扑排序、生成完整 `.task` 段 |
| **执行** | `.atxe` | — | 运行时加载，按调度策略执行 |

### 7.2 每个 Task 是独立链接单元

Task A 和 Task B 各自产出独立的 `.atxe`，可以有完全不同的依赖版本。它们共享同一个源码仓库的 TOOLS 和 WORKS 定义，但链接阶段各自解析各自的依赖树。

## 8. 闭包捕获语义

`do` 匿名函数**按值捕获**外部变量：

```
x : int = 10
f = do (y : int) : int {
    x + y       # x 按值捕获，捕获时 x = 10
}
x = 20          # 后续修改不影响 f 中的 x
result = f(5)   # result = 15
```

按值捕获的原因：
- VM 寄存器模型不提供引用语义
- 避免闭包逃逸后的悬垂引用
- 编译器可静态确定捕获列表，生成高效的寄存器复制序列

## 9. 文件元信息：`#! !#`

`.atx` 源文件中的 `#! ... !#` 块注释是**纯文本信息卡片**，编译进 IR 的元信息段以供追溯。

它**不包含任何影响编译或运行时行为的配置**——那是 `atomix.toml` 的职责。它的作用是：当你在生产机上拿到一个 `.atxe` 文件，可以追溯到它是谁写的、干什么用的、什么版本。

```atomix
#!
# 此文件中所有字段均为可选。不影响任何编译或运行时行为。
# 填写的内容编译进 IR 以供追溯。

file        = "tasks/cleanup.atx"
summary     = "Daily data cleanup pipeline"
description = "多行详细描述。"

version     = "1.2.0"
author      = "infra-team <infra@example.com>"
license     = "MIT"

category    = "data-pipeline"
tags        = ["cleanup", "etl", "daily"]

usage       = "atx run data_cleanup"
changelog   = "1.2.0: 新增去重逻辑"

custom_jira = "INFRA-1234"
!#
```
