> 架构版本: v0.1 (设计阶段)
> 适用范围: TASK 语法域
> 配套文档: [01-总纲与哲学.md](../01-总纲与哲学.md)、[通用语法.md](通用语法.md)

## 1. 语法域概述

TASK 区是 Atomix 的编排核心——引用 INPUT 常量，调用函数，派发 WORKS 实例，标注产出变量。

> **TASK ≈ main 函数**。它是编排入口，不是定义场所。

### 1.1 能做什么

| 能力 | 说明 |
|------|------|
| `CALL` | 调用 TOOLS 函数或 WORKS 模板（Step 执行） |
| `IF`/`FOR` | 条件分支与循环 |
| `WAIT` | 派发 WORKS 实例 |
| `GOOUT` | 标注产出变量 |
| 变量声明 | 创建局部变量 |
| 值表达式 | 逻辑运算、算术运算 |

### 1.2 不能做什么

| 禁令 | 原因 |
|------|------|
| **不能定义函数** | 函数定义归 TOOLS 区 |
| **不能声明类型** | 类型定义归 TOOLS 区 |
| **不能声明异常** | 异常定义归 TOOLS 区 |
| **不能声明 WORKS** | 任务模板归 WORKS 区 |

TASK 是纯编排层。你在 TASK 里写函数定义，等于在 main 里定义类——语法不拦你（无意义不危险就不拦），但工程规范不建议。

### 1.3 执行与生命周期

TASK 执行完毕后，所有非 `GOOUT` 标注的局部变量销毁回收，`GOOUT` 变量等待 OUT 区交付。

引用方向：TASK 可引用 TOOLS、INPUT、WORKS。不可引用 OUT。

## 3. CALL

### 3.1 模板

```
<CALL声明>  : CALL [ <输入> <赋值箭头> ] <执行体> [ <赋值箭头> <输出> ] [ <尾随关键词> ]

<执行体>    : <标识符> ( <参数列表> )
<参数列表>  : <值表达式> [ , <值表达式> ]*
```

`CALL` 和 `<执行体>` 是必需项。**输入和输出互斥**——要么左边写输入，要么右边写输出，不可两者兼得。

> **CALL 的使用范围**：CALL 是 TASK 编排的核心工具，原则上建议在 TASK 区使用。语法上不限制在其他区使用——行为通顺，但内容不理解，尊重。如果你在 TOOLS 或 WORKS 中写 CALL，语法不拦你。

### 3.2 CALL 即 Step

CALL 不只是函数调用——它在执行系统中代表一个**Step**（执行步骤）。

- **Step 名**从执行体自动派生（`CALL process(data)` → step 名 `process`）
- **日志按 Step 聚合**：Step A 期间产生的所有日志归 A，Step B 归 B，互不混淆
- **进度汇报**：运行时可以根据 Step 报告当前执行进度
- 没有 CALL 前缀的直接调用（`func()`）只是普通表达式求值，不构成 Step，不参与日志聚合和进度跟踪

```
CALL fetch_data()      # Step: fetch_data — 有日志聚合、进度追踪
x = compute(y)         # 普通调用，不构成 Step
```

这意味着 CALL 是 TASK 区中**结构化执行的最小可观测单元**。

> **不要滥用 CALL。** CALL 是 Step 级别的高级设施——全上 CALL 会导致每个函数都成为 Step，日志被切得稀碎。只有在需要 Step 级别观测的地方才用 CALL，普通函数调用直接写函数名即可。
>
> **CALL 的使用范围**：CALL 是 TASK 编排的核心工具，原则上建议在 TASK 区使用。语法上不限制在其他区使用——行为通顺，但内容不理解，尊重。

### 3.3 两种书写风格

**风格一：变量在前（熟悉派）**

```
CALL raw = PROCESS(data) TRY {}
```

输入变量在左，`=` 赋值到执行体。先看到数据从哪来，再看到怎么处理。

**风格二：输出在后（直观派）**

```
CALL PROCESS(data) => result TRY {}
```

执行体在前，`=>` 输出到结果变量。先看到处理逻辑，再看到结果去哪。

### 3.4 不带输入或输出

```
CALL PROCESS(data)          # 无输入、无输出——纯执行，不关心数据
CALL log("done") TRY {}     # 无输入输出但有尾随
```

### 3.5 赋值箭头语义

与通用语法 §8 一致，取决于箭头出现在输入侧还是输出侧：

| 箭头 | 输入侧（左边） | 输出侧（右边） |
|------|---------------|---------------|
| `=` | COPY 输入到执行体 | COPY 结果到变量 |
| `=>` | MOVE 输入到执行体 | MOVE 结果到变量 |
| `<=` | — | 反向 MOVE |

## 4. IF

```
<IF声明>    : IF <值表达式> { <TASK声明>+ }
```

条件为 `true` 时执行块内代码。`<值表达式>` 为通用语法 §7 定义的值表达式：

```
IF a > b {
    CALL process(a)
}
IF x == 10 and y != 0 {
    CALL compute(x, y)
}
```

---

## 5. FOR

```
<FOR声明>   : FOR <值表达式> { <TASK声明>+ }
```

条件为 `true` 时持续执行块内代码，每次执行后重新求值：

```
i = 0
FOR i < 10 {
    CALL process(items[i])
    i = i + 1
}
```

## 6. WAIT

WAIT 派发 WORKS 区定义的任务模板到执行队列。派发后数据通过赋值箭头自然回流，不需要单独的 JOIN 操作。

### 6.1 模板

```
<WAIT声明>  : WAIT [ <输入> <赋值箭头> ] <模板名>
                [ ( <覆盖参数列表> ) ]
                [ <赋值箭头> <输出> ]
                [ <尾随关键词> ]

<覆盖参数列表> : <常量名> = <值表达式>
                [ , <常量名> = <值表达式> ]*
```

### 6.2 默认行为（不填参数）

不填参数时，WAIT 走 INPUT 的完整数据流——从源获取数据、执行全部流程，然后传入 WORKS 模板：

```
WAIT data_pipeline => result
# 自动执行 INPUT 的所有源声明，取数据后传给 data_pipeline
```

### 6.3 参数覆盖（最有意思的部分）

当你填了参数，对应常量的**源获取逻辑被顶掉**——不跑那个源了，直接用你填的值：

```
WAIT data_pipeline (RAW = "custom_data") => result
# RAW 原本从 HTTP 获取，现在直接用 "custom_data"
# 其他没覆盖的常量仍走默认源
```

**规则：**
- 只能覆盖**顶层 INPUT 常量**（`INPUT {}` 直接子级），嵌套内的局部变量不可覆盖
- 没填的常量照常走源获取逻辑
- 覆盖的值可以是字面量、表达式、或其他变量引用

```
WAIT process_data (
    RAW = precomputed,          # 用变量覆盖
    TIMEOUT = DEFAULT,          # 用 DEFAULT 关键字恢复默认
    MAX_RETRY = 3               # 用字面量覆盖
) => result TRY {}
```

## 7. 示例

```
TASK :
    # 简单调用
    CALL fetch_data()
    
    # 输入→处理→输出
    CALL RAW => PROCESS(RAW) => result
    
    # 带尾随 TRY
    CALL fetch_data() TRY ISTIMEOUT == 10s {
        print(ISTIMEOUT)
    }
    
    # 用 GOOUT 标注产出变量
    CALL compute(INPUT : RAW) => GOOUT result : i32
```

## 8. 非法情况

| 情况 | 说明 |
|------|------|
| CALL 内的 `<执行体>` 非标识符 | `CALL 42()` |
| 尾随关键词用于非 CALL 场景 | `x = 42 TRY`（已在通用语法中禁止） |
| 引用未定义的函数或工具 | `CALL undefined_func()` |
