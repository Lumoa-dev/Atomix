# Atomix 类型系统

> 架构版本: v0.1 (设计阶段)
> 最后更新: 2026-07-15
> 所属体系: 语法设计

---

## 1. 设计原则

### 1.1 核心原则

| 原则 | 含义 |
|------|------|
| **强类型** | 编译期类型检查，不存在隐式类型转换。运行时类型完全确定 |
| **源头类型确定** | 每个类型信息链都有一个显式的源头锚点，类型从源头沿数据流动传播到所有变量 |
| **渐进式标注** | 支持显式（C 风格 `int x`）和隐式（Python 风格 `x = 42`）两种声明，编译器统一处理 |
| **无对象类型** | 无 GC、无装箱、无运行时类型标识 |

### 1.2 什么是"源头类型确定"

类型信息的传播不是凭空推导的，而是有一个**明确的源头**——要么是你写的类型标注，要么是编译器认识的字面量/关键字类型。类型从源头出发，沿着数据流动链（赋值、箭头、函数调用）传播到所有相关变量。

```
                源头类型（显式标注 / 字面量 / 关键字）
                       │
                       ▼
              ┌──────────────────┐
              │  类型传播链        │
              │  · 赋值 → 变量    │
              │  · 箭头 → 目标    │
              │  · 函数返回 → 结果 │
              │  · 运算符 → 产出  │
              └──────────────────┘
                       │
                       ▼
              所有变量编译期确定类型
```

**关键规则：** 每条类型信息链必须有且仅有一个源头。编译器从源头出发，沿数据流传播类型。若某条链找不到源头（即没有任何标注、字面量或已知类型信息），编译报错"无法确定类型"。

### 1.3 四种声明风格

| 风格 | 写法 | 类型来源 | 适用场景 |
|------|------|----------|----------|
| 显式类型（C 风格） | `int count = 42` | 标注字面量 `int` 是源头 | 边界接口、函数签名、需要强调类型的场景 |
| 隐式类型（Python 风格） | `count = 42` | `42` 字面量 `int` 是源头 | 局部变量、中间结果、类型一目了然的场景 |
| 注解类型（Python 注解风格） | `count: int = 42` | 标注 `int` 是源头，初始值必须匹配 | 需要显式类型但习惯变量名在前的场景 |
| 关键字声明（DSL 风格） | `OUT : result : list` | 关键字或标注是源头 | INPUT/OUT 区数据流动声明 |

四种风格在文件中可以混合使用，最终效果一样——编译期全部确定类型。

```
// 显式 —— int 是源头
int count = 42

// 隐式 —— "hello" 字面量的 string 是源头
name = "hello"

// 混合 —— result 的类型从 CALL 返回类型的源头来
CALL : fetch_data(INPUT : URL) => result   // result 类型 = fetch_data 的返回类型
```

## 2. 源头类型总表

### 2.1 类型即源头

以下场景中的类型是**天然的源头**：

| 源头种类 | 示例 | 确定类型 |
|----------|------|----------|
| 整数字面量 | `42` `-7` `0xFF` | `int` |
| 浮点字面量 | `3.14` `-0.5` `1e10` | `float` |
| 字符串字面量 | `"hello"` `"你好"` | `string` |
| 布尔字面量 | `true` `false` | `bool` |
| 空值字面量 | `null` | `null`（按上下文确定具体类型） |
| 列表字面量 | `[1, 2, 3]` | `list[int]` |
| 字典字面量 | `{"k": "v"}` | `dict[string, string]` |
| 显式类型标注 | `int x` `=> raw : bytes` | 标注所指定的类型 |
| 函数返回类型 | `int : fn add(...)` | 声明的返回类型 `int` |
| 函数参数类型 | `(string input)` | 声明的参数类型 `string` |
| IO 关键字类型 | `HTTP` `JSON` `CSV` 等 | 关键字自带的产出类型 |
| INPUT 常量声明 | `=> RAW : bytes` | `bytes` |
| CONST 声明 | `CONST int MAX = 10` | `int` |

### 2.2 类型总表

| 类型 | 说明 | 示例字面量 | 可作为源头 |
|------|------|-----------|-----------|
| `int` | 有符号整型（64位） | `42` `-7` `0xFF` | ✅ |
| `float` | 浮点型（64位） | `3.14` `-0.5` `1e10` | ✅ |
| `string` | 字符串 | `"hello"` `"你好"` | ✅ |
| `bytes` | 字节序列 | `0x68656C6C6F` | ✅ |
| `bool` | 布尔值 | `true` `false` | ✅ |
| `list[T]` | 列表（元素类型 T） | `[1, 2, 3]` | ✅ |
| `dict[K, V]` | 字典（键类型 K，值类型 V） | `{"k": "v"}` | ✅ |
| `void` | 无返回值，仅用于函数返回类型 | — | ❌ |
| `null` | 空值 | `null` | ❌（按上下文定） |

#### int

有符号 64 位整型。支持十进制和十六进制字面量。

```
int a = 42
int b = -7
int c = 0xFF          // 255
```

**支持运算符：** `+` `-` `*` `/` `%` `&` `|` `^` `<<` `>>` `~` `==` `!=` `<` `>` `<=` `>=`

#### float

64 位浮点型（IEEE 754）。

```
float a = 3.14
float b = -0.5
float c = 1e10        // 科学计数法
```

**支持运算符：** `+` `-` `*` `/` `==` `!=` `<` `>` `<=` `>=`

> `%` `&` `|` `^` `~` `<<` `>>` 等整型运算符不适用于 float。

#### string

字符串，双引号包裹。支持转义序列。

```
string a = "hello"
string b = "line1\nline2"
string c = "path\\to\\file"
```

**支持运算符：** `+`（拼接）、`==` `!=`

#### bytes

字节序列。通常用于二进制数据。

```
bytes a = 0x68656C6C6F    // "hello" 的十六进制
bytes b = to_bytes("hello")
```

#### bool

布尔值。仅两个取值：`true` `false`。

```
bool a = true
bool b = (count > 0)
```

**支持运算符：** `==` `!=` `&&` `||` `!`

#### list[T]

动态数组，元素类型为 T。

```
list[int] a = [1, 2, 3]
list[string] b = ["a", "b", "c"]
```

元素类型一致时编译器推导类型参数；混合类型元素需要显式标注：

```
list a = [1, "hello", true]          // ❌ 元素类型不一致，推导失败
list b = [1, 2, 3]                   // ✅ 推导为 list[int]
list[str] c = ["a", "b", "c"]       // ✅ 显式标注
```

通过索引访问：

```
int first = a[0]
```

**支持运算符：** `==` `!=`

#### dict[K, V]

键值映射。键 K 通常为 string 或 int，值 V 可为任意类型。

```
dict[string, int] a = {"name": "atomix", "version": 1}
dict[int, string] b = {1: "one", 2: "two"}
```

通过键访问：

```
string name = a["name"]
```

**支持运算符：** `==` `!=`

#### void

`void` 不是一个"值"，而是"没有值"的标记。仅用于函数/方法的返回类型声明，表示该函数不返回值。

```
void : fn log_message(string msg) {
    print(msg)
    // 没有 return
}
```

`void` 类型不能声明变量，不能用作参数类型，不能用作列表/字典的元素类型。

#### null

`null` 表示"空值"，可以赋值给引用类型（string、list、dict）。

```
string a = null
list b = null
dict[string, int] c = null
```

`null` 不可赋值给值类型（int、float、bool、bytes）：

```
int a = null         // ❌ 编译错误
string b = null      // ✅
```

判断是否为空：

```
if (a == null) { ... }
if (a != null) { ... }
```

## 3. 源头类型传播规则

### 3.1 传播路径

类型从源头出发，沿以下路径传播：

| 路径 | 规则 | 示例 |
|------|------|------|
| **变量声明** | 显式标注的类型是源头；无标注时从右侧初始化表达式获取类型 | `int x = 42` → x 是 int；`x = 42` → 42 是源头，x 推导为 int |
| **箭头流动** | 箭头左侧的类型传播到右侧变量 | `HTTP : "url" => RAW : bytes` → RAW 是 bytes |
| **函数返回** | `return x` 的类型必须匹配函数声明的返回类型 | `int : fn add(...) { return x }` → x 必须是 int |
| **函数参数** | 调用时传入的实际参数类型必须匹配声明的参数类型 | `fn process(string input)` → 传入的实参必须是 string |
| **赋值** | 右侧值类型传播到左侧变量 | `a = b` → b 的类型覆盖 a（a 必须已声明类型或被推导出） |
| **运算符** | 操作数类型决定结果类型 | `a + b` → 若 a、b 都是 int，结果为 int |
| **比较运算** | 任何类型比较结果恒为 bool | `a > b` → bool，无论 a、b 类型 |
| **索引访问** | list 索引返回元素类型，dict 索引返回值类型 | `a[0]` → int（若 a 是 list[int]） |
| **FOR 迭代** | 迭代变量的类型 = 集合的元素类型 | `for (item in items)` → item 的类型 = items 的元素类型 |
| **CALL 产出** | `CALL : func() => x` → x 的类型 = func 的返回类型 | `CALL : fetch() => result` → result 类型 = fetch 的返回类型 |
| **IO 关键字** | 关键字自带产出类型 | `HTTP` → bytes，`JSON` → dict，`CSV` → list[dict] |

### 3.2 类型传播链示例

```
// 链①：字面量 → 变量 → 运算符 → 变量
count = 42                // 42(int) → count(int)
result = count + 8        // count(int) + 8(int) → result(int)

// 链②：显式标注 → 函数调用 → 变量
int : fn parse(string raw) { ... }   // 返回类型 int 是源头
CALL : parse(input) => value         // value(int) 从 parse 的返回类型来

// 链③：INPUT 常量 → TASK 引用
INPUT :
    HTTP : "url" => RAW : bytes       // bytes 是源头
TASK :
    CALL : process(RAW) => result     // RAW(bytes) → process 的第一个参数必须是 bytes

// 链④：FOR 集合 → 迭代变量
list[int] items = [1, 2, 3]          // list[int] 是源头
for (item in items) {                // item(int) 从 items 的元素类型来
    CALL : handle(item)              // item(int) → handle 的参数必须是 int
}
```

### 3.3 类型确定性的保证

| 阶段 | 保证 |
|------|------|
| **编译期** | 所有变量和表达式的类型已确定。不存在"运行时才知道类型"的情况 |
| **类型检查不通过** | 编译失败，报错位置 + 预期类型 vs 实际类型 |
| **类型检查通过** | 运行时类型与编译期类型完全一致，不会出现类型不匹配 |

### 3.4 无法推导的场景

编译器遇到无法确定源头类型的变量时，报编译错误：

```
// 无法推导：count 没有显式类型，右侧也不是字面量/已知类型
CALL : fetch() => data
count = data            // ❌ data 的类型需要从 fetch 的返回类型确定
                        //    但如果 fetch 的返回类型也没标注 → 报错

// 可以推导：fetch 有返回类型标注
int : fn fetch() { ... }
CALL : fetch() => data   // data → int
count = data             // count → int（从 data 传播）
```

## 4. 类型标注语法

### 4.1 显式类型声明（C 风格）

```
<类型> <变量>
<类型> <变量> = <初始值>
```

```
int count
string name = "atomix"
bytes raw
list[int] items = [1, 2, 3]
dict[string, string] config
```

类型关键字在前，是类型的显式源头。

### 4.2 隐式类型声明（Python 风格）

```
<变量> = <初始值>
```

```
count = 42               // int
name = "atomix"          // string
items = [1, 2, 3]        // list[int]
```

类型由右侧初始值中的源头类型推导而来。

### 4.3 关键字声明（DSL 风格）

```
<关键字> : <变量> [: <类型>]
```

```
OUT : result : list[int]
OUT : batch_data : list
```

关键字在前，类型可选。省略时由关键字自带产出类型推导。

### 4.4 类型标注（箭头后）

```
... => <变量> : <类型>
```

类型标注不能独立出现，必须附着在有数据流动的上下文中：

```
=> raw : bytes          // ✅ 箭头后标注
JOIN : x, y => acc : list  // ✅ 聚合结果附带类型
raw : bytes              // ❌ 无数据变化来源，标注无意义
```

## 5. 类型转换

内置函数提供显式类型转换：

| 函数 | 源→目标 | 说明 |
|------|---------|------|
| `to_int(x)` | string/float → int | 字符串解析，浮点数截断 |
| `to_float(x)` | int/string → float | 整数提升，字符串解析 |
| `to_string(x)` | 任意 → string | 通用转字符串 |
| `to_bytes(x)` | string → bytes | 字符串编码为字节 |

> Atomix **不提供**隐式类型转换。所有类型转换必须通过显式函数调用完成。

## 6. 常量

常量用 `CONST` 关键字声明，一经赋值不可修改：

```
CONST <类型> <变量名> = <值>
```

```
CONST int MAX_RETRY = 10
CONST string APP_NAME = "Atomix"
CONST float PI = 3.14159
CONST bool DEBUG = true
```

**规则：**
- 常量声明时**必须标注类型**（const 类型是编译期常量传播的依据）
- 常量不可被赋值（编译期检查）
- 常量作用域为当前区
- 常量名推荐全大写蛇形命名（`MAX_RETRY`、`APP_NAME`）

## 7. 类型检查规则

| 场景 | 行为 |
|------|------|
| 赋值 `int a = "hello"` | ❌ 类型不匹配，编译错误 |
| 赋值 `a = "hello"` 且 `a` 已为 int | ❌ 类型不匹配，编译错误 |
| 赋值 `int a = null` | ❌ null 不可赋值给值类型 |
| 赋值 `string a = null` | ✅ null 可赋值给引用类型 |
| 赋值 `int a = 42` | ✅ 类型匹配 |
| 函数返回类型不匹配 | ❌ 编译错误 |
| 函数参数类型不匹配 | ❌ 编译错误 |
| 隐式类型转换 | ❌ 必须显式调用转换函数 |
| 运算符类型不匹配（`int + float`） | ❌ 编译错误（需显式转换） |
| 索引越界（编译期可确定的越界） | ❌ 编译错误 |
| 变量未声明直接使用 | ❌ 编译错误 |
| 类型链断裂（找不到源头） | ❌ 编译错误 |

## 8. 设计边界

### 8.1 为什么没有枚举类型？

枚举的应用场景在任务编排 DSL 中较窄（状态码、错误分类等均可由字符串常量或 int 常量替代）。增加 enum 语法带来的复杂度超过其收益。后续若需求明确可补充。

### 8.2 为什么没有类型别名？

当前类型系统较简单，`list[int]`、`dict[string, string]` 等直接书写即可满足需求。类型别名在类型复杂度提升后再考虑加入。

### 8.3 为什么没有函数类型？

函数不作为一等值传递，不存在高阶函数场景。函数类型仅在"函数作为参数"时需要，当前设计不支持此模式。

### 8.4 为什么没有联合类型（union）？

联合类型的主要场景在错误处理和状态机中，Atomix 通过 `CALLIF` + `IS*` 状态字处理错误，无需 union 类型表达"成功或失败"。
